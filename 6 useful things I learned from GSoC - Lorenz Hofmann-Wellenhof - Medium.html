6 useful things I learned from GSoC

Lorenz Hofmann-Wellenhof
Aug 23 · 5 min read

Google Summer of Code (GSoC) is a global program focused on bringing more student developers into open source software development. Students work with an open source organization on a 3 month programming project during their break from school.

The summer of 2019 I spent coding for the ‘API Client Tools at Google’ organization. I was lucky enough to get chosen by my Mentors Tim Burks and Noah Dietz (software engineers at Google) for this project.

Things I learned

Golang & environment: Before GSoC I did not write a single line of code with Go. At the beginning I was a bit skeptical, because it felt like C and I never did any serious programming with C, but I started to love Go after 5 weeks in, because everything just works so smoothly.

gRPC: Just as with Go, before GSoC I had no clue what gRPC is or that it even existed. I guess that 90%+ of our daily internet transactions are based on REST. gRPC is an alternative approach. It is an open source version of Stubby (Google’s internal API calling system) based on HTTP/2 and has therefore many advantages over the typical REST approach which is based on HTTP/1. However, since gRPC is not that widely used, documentation might be scarce.

Protocol Buffers: Protocol buffers are Google’s language-neutral, platform-neutral, extensible mechanism for serializing structured data — think JSON, but smaller, faster, and simpler. And again: I had no idea that such a mechanism exists before GSoC. Protocol buffers go hand in hand with gRPC. Isn’t it funny that so many people have the goal to get hired by Google and have no clue about the internal technologies they use? I guess that RPC, Protocol Buffers, and Go are the basics that you will learn in the first couple months while you work there.

Improved my coding and software architecture skills: Ok, this is probably the most important one. Even though it is difficult to measure the progress, I definitely became better at connecting different components and removing overall complexity. Additionally, thanks to Noah I became more aware of what is happening outside of my if-else statement, because he was very, very, persistent during code reviews.

Time management: During the entire three months of the program I never really felt stressed. Tim and Noah were always super chill. I had enough time to work a bit on my master thesis, do sports, and enjoy the summer in Austria with my friends. Overall I worked 23 hours per week. However, the time for writing the project proposal (and getting to know the technologies) is not included. I would estimate additional 20h per week for four weeks.

Basics of envoy: While creating an end-to-end tutorial on how to set up a gRPC API with HTTP transcoding and envoy, I learned the basics of envoy. Originally built at Lyft, it is a high performance open source service proxy, designed to work with microservices.

Challenges

I think the most difficult challenge was to come into this ‘self management state’ where you have enough knowledge of the used technologies to come up with new tasks and ideas to improve the project.

There were a couple of days were I started working and was like: ‘Ok, I am not entirely sure what to do today.’ I believe the best way to handle this situation is to start investigating related aspects to your project: I started browsing related GitHub pages, looked through my code and thought about what could be improved, checked Twitter/GitHub from my mentors, and read about the technologies and their concepts that are used for the project. At some point you realize what your next tasks should be and start implementing them.

I never really wanted to ask my mentors: ‘What should I do next?’, because in my opinion a somewhat decent software engineer comes up with the tasks on his own. However, every bit of information from my mentors was extremely crucial to the project and improved the overall quality a lot. We communicated mainly over Google Docs where I listed my daily accomplishments, my tasks for tomorrow, and my current problems:

This approach is very similar to a daily stand up meeting from agile methodologies. Whenever Tim or Noah wanted to add additional input they wrote a comment. I can definitely recommend this approach for remote projects such as GSoC.

Concepts of the project

The main idea of the project is to convert an OpenAPI description into a description of a gRPC service that can be used to implement that API using gRPC-JSON Transcoding.

It works the following way:

  1. gnostic converts an OpenAPI description (bookstore.yaml) into an equivalent Protocol Buffer representation (bookstore.pb).
  2. The surface model of gnostic then converts the Protocol Buffer representation into plugin-friendly data structures.
  3. gnostic-grpc (a plugin for gnostic) converts the data structures into a FileDescriptorSet (bookstore.descr). A FileDescriptorSet describes the messages and methods of a Protocol Buffer definition file (bookstore.proto).
  4. In order to generate the final Protocol Buffer definition file (bookstore.proto) from the FileDescriptorSet, jhump’s protoreflect library is used.
